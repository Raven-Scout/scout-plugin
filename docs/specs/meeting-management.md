# Meeting Management System + Inbox for Scout

## Context

Scout currently handles meetings reactively — it reads calendar events and mentions them in action items, but doesn't generate prep docs, track meeting history, or synthesize post-meeting notes. Users often have many recurring meetings and take notes manually or get transcripts from Drive. They need a system where prep, notes, and synthesis flow through a single file per meeting instance, with recurring meetings having a persistent "home" that tracks running themes and decisions.

Additionally, users need a single quick-capture file (`inbox.md`) where they can dump notes, reminders, and ideas at any time. {{INSTANCE_NAME}} triages these into the right places (meeting prep, action items, project files, personal tasks) during each run.

## Design: Meeting-Centric Folders

### Folder Structure

```
meetings/
  meetings.md                         # Registry/index of all tracked meetings
  team-standup/
    team-standup.md                   # Home: purpose, attendees, running themes, decisions
    2026-04-22.md                     # Dated file: prep → notes → synthesis
    2026-04-23.md
  weekly-1on1/
    weekly-1on1.md
    2026-04-21.md
  project-sync/
    ...
```

Remove the empty `action-items/meeting-prep/` directory — prep lives in the dated files.

### Three-Phase Dated File

One file per meeting instance, evolving through three states:

1. **Prep** (generated by {{INSTANCE_NAME}} before meeting): Previous session context, open action items, project status, suggested talking points
2. **Notes** ({{USER_NAME}} writes during/after): Free-form in a Notes section, or left blank for transcript
3. **Synthesis** ({{INSTANCE_NAME}} fills post-meeting): Merges manual notes + Drive transcript, extracts decisions and action items, propagates to project files and home file

### Home File

Each recurring meeting has a home file with:
- `calendar_match` YAML frontmatter for title-based matching (e.g. `["team standup", "Team Standup"]`)
- Regular attendees, related projects
- **Running Themes** — persistent topics across sessions (updated by {{INSTANCE_NAME}} after each synthesis)
- **Key Decisions** table with wikilinks to dated files and project files
- **Recent Sessions** table

### Calendar Matching

Match by normalized title substring using `calendar_match` patterns. More robust than recurring event IDs which break on reschedule/recreate. Unmatched recurring events get flagged for {{USER_NAME}} to configure.

### Action Items Integration

Meeting prep is surfaced in the daily action-items file in two ways:

**1. Today's Meetings table with prep wikilinks:**

```markdown
## 📅 Today's Meetings

| Time | Meeting | Prep | Status |
|------|---------|------|--------|
| 09:30 | Team Standup | [[team-standup/2026-04-21|prep]] | upcoming |
| 14:30 | Weekly 1:1 | [[weekly-1on1/2026-04-21|prep]] | upcoming |
| 15:00 | Project Sync | [[project-sync/2026-04-21|prep]] | upcoming |
```

After meetings complete, consolidation updates status to `done` and links to the synthesis.

**2. Preparation action items reference the prep file:**

```markdown
- **Prepare PR #477 summary for project sync** — attendee will ask about status
  - Source: Calendar + GitHub
  - Context: [[project-sync/2026-04-21]]
```

**3. Post-meeting action items trace back to the meeting:**

After synthesis, extracted action items include the meeting as source:

```markdown
- **Follow up with infra team on dependency** — owner committed to this
  - Source: [[project-sync/2026-04-21]] (synthesis)
  - Context: [[project-name]]
```

This creates a full chain: action-items <-> meeting prep/synthesis <-> project files.

### Inbox — Quick Capture

A single file `inbox.md` for dumping anything, anytime:

```markdown
# Inbox

- Ask team lead about timeline in next project sync
- Remember to mention pricing update to stakeholder
- Idea: could we use the ontology parser for meeting attendee matching?
- Personal: renew passport before June trip
```

**Processing (every briefing + consolidation run):**
1. {{INSTANCE_NAME}} reads all entries
2. Routes each to the right place:
   - Meeting-related note -> target meeting's home file under `## Next Session Notes`
   - Action item -> today's action-items file
   - Personal task -> create/update entity in `knowledge-base/personal/`
   - Project idea/note -> relevant project file or `knowledge-base/research-queue.md`
   - Ambiguous -> keeps in inbox with a `[needs context]` flag for {{USER_NAME}} to clarify
3. Removes processed entries from inbox (git preserves history)
4. If the inbox is empty, skip silently

**Routing heuristics:**
- Mentions a meeting name or person associated with a meeting -> meeting prep
- Starts with "personal:" or mentions non-work topics -> personal task
- References a project name or Linear issue -> project file
- Contains "idea:", "research:", "explore:" -> research queue
- Everything else -> action item (safest default)

{{USER_NAME}} only ever opens one file. {{INSTANCE_NAME}} does the routing.

## Implementation Steps

### Step 0: Create inbox.md

Create `inbox.md` with a simple header. Add `[[inbox]]` to `knowledge-base/knowledge-base.md` navigation.

### Step 1: Create meeting structure + registry + home files

**Files to create:**
- `meetings/meetings.md` — registry with table of all recurring meetings
- Home files for recurring meetings (derived from calendar data in action-items)
- Seed home files with attendees, related projects, running themes from existing KB content

**Files to modify:**
- `knowledge-base/knowledge-base.md` — add `[[meetings]]` to navigation table

**Files to remove:**
- `action-items/meeting-prep/` — replaced by meetings/

### Step 2: Add "Inbox Processing" to SKILL.md (Briefing + Consolidation)

Add as the first sub-step of both briefing and consolidation modes. Logic:
1. Read `inbox.md`
2. If empty, skip
3. For each entry, classify using routing heuristics and route to target
4. Remove processed entries, keep `[needs context]` items
5. If any entries routed to meeting home files, those notes will be picked up by meeting prep generation

### Step 3: Add "Meeting Prep Generation" to SKILL.md (Briefing mode)

Insert after calendar query, before writing action items. Logic:

1. Read `meetings/meetings.md` for tracked meetings
2. For each meeting today, match to folder via `calendar_match`
3. If matched and no dated file exists, create it with Prep section:
   - Read home file for attendees, themes, projects
   - Read last 1 dated session file for open action items
   - Query project files, Linear issues, Slack threads for current context
   - Generate "Suggested Talking Points"
4. Unmatched recurring events -> add to registry with `[new]` flag, create folder
5. One-off meetings with 2+ attendees + project link -> create folder too

**Update the action-items format:**
- Replace the current "Today's Schedule" section with a "Today's Meetings" table that includes a Prep column with `[[wikilinks]]` to the dated session files
- Any preparation-specific action items (e.g. "review PR before sync") must include `Context: [[meeting-slug/YYYY-MM-DD]]` linking to the prep file

### Step 4: Add "Post-Meeting Processing" to SKILL.md (Consolidation mode)

Insert as Phase 2.5 (after inbound scan, before per-item reconciliation). Logic:

1. Check calendar for meetings ended since last run
2. For each completed meeting with a folder:
   a. Check dated file for manual notes in Notes section
   b. Search Drive for transcripts matching meeting title + date
   c. Merge notes + transcript into Synthesis section
   d. Propagate decisions -> home file Key Decisions + project files Decisions Made
   e. Propagate action items -> daily action-items file (with standard cross-check), each citing `Source: [[meeting-slug/YYYY-MM-DD]]`
   f. Update Running Themes in home file
   g. Update the Today's Meetings table in action-items: set status to `done`, link synthesis
3. If no notes or transcript exist, flag as "no notes captured"

### Step 5: Update DREAMING.md scoring

- Add meeting home files to KB scoring dimensions (staleness, gaps, structural integrity)
- Freshness: home files = 7 days, registry = every briefing, dated files = immutable after synthesis
- Inactive series (>30 days no occurrence) -> candidate for `meetings/archived/`

### Step 6: Migrate existing meeting content

- Move meeting rhythm from project files to respective home files (keep wikilinks)
- Project files' "Upcoming Meetings" sections gain wikilinks to meeting folders

### Step 7: Commit

```bash
cd {{SCOUT_DIR}} && git add -A && git commit -m "Add meeting management system: registry, home files, prep/synthesis workflow with action-items integration"
```
